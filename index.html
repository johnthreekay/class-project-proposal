Class Project Proposal

Product Name

Ryokan

Team Members

John Keslin

Application Overview

Ryokan is a self-hosted anime PVR. You tell it what shows to track, and it handles the rest: pulls metadata from AniList, searches Nyaa for releases, scores them, sends the best match to qBittorrent, and imports the finished download into a Jellyfin library. An RSS loop picks up new episodes as they air, and a separate sweep quietly re-evaluates older episodes when better encodes show up.

I built it because Sonarr, which is what most people use for automated TV downloads, is bad at anime specifically. It works fine for currently-airing simulcasts, but its interactive search hangs on finished series because it looks for SXEXX-style episode numbers that anime releases almost never use. That's a fundamental design mismatch, and every anime-focused "fix" I tried was either a thin Sonarr wrapper with the same core problem or a GUI over yt-dlp with no library management. So I wrote my own.

The one non-obvious design decision is that Ryokan pretends to be Sonarr and Radarr over HTTP, so Seerr (the request-management frontend I already use for movies and Western TV) can talk to it without any modifications on Seerr's side. That means requesting an anime goes through the exact same flow as requesting a movie: search in Seerr, click request, it shows up in Ryokan, Ryokan grabs it.

Technology Overview

The whole backend is Rust on Axum 0.8 with Tokio, sqlx against SQLite (runtime-checked queries, no compile-time SQL macros), and Askama for templates. Those get compiled into the binary so there's nothing to serve separately. The frontend is one stylesheet and vanilla JS inlined into the templates; no framework, no bundler, no build step outside cargo. OpenAPI docs come from utoipa, structured logging from tracing, HTTP from reqwest, and Nyaa parsing from scraper. The whole thing ships as a multi-arch Docker image (amd64 and arm64) built by GitHub Actions, and I run it behind a Cloudflare Tunnel on a Linux box at home.

Feature List

  • Finding and grabbing releases. The library page is a cover grid of everything you're tracking. Click a show, hit a button, and Ryokan searches Nyaa and shows you ranked results. You can one-click grab a specific release (interactive search), or let the automation pick based on your quality preferences. RSS picks up new episodes as they air; a separate scheduled sweep re-evaluates older ones for quality upgrades, but only up to a cutoff tier you set, so it doesn't chase releases forever.
  • Scoring. Every release gets a score from three sources that stack: Sonarr-style Custom Formats (the TRaSH Guides anime defaults ship as a one-click install and are what most users will want), a classification pipeline that figures out whether a file is actually BluRay / Web / HDTV and at what resolution, and optional matching against SeaDex, which is a community-curated list of "this is the best release of this show." SeaDex is available either as a settings toggle or as a Custom Format spec I added specifically for Ryokan.
  • The classification pipeline. Deciding what source a release really is turns out to be genuinely hard. The filename lies sometimes (a "BD" tag on a Web-DL rip), release groups have house styles you can use as priors, and some things are only knowable after you've downloaded the file and probed its audio codecs. I combined six signals (filename tokens, Nyaa's description body, the release date versus the show's airing schedule, a release-group identity map, the downloaded directory structure, and ffprobe output) into one verdict with a confidence score. When confidence is low the episode lands in a "needs review" queue. When you manually override a classification, Ryokan remembers the (group, source) pair and starts suggesting it could be promoted into the identity map if it sees enough repeats.
  • Post-processing. When a torrent finishes, Ryokan reads it out of qBittorrent and imports it into the library. Default mode is hardlink, which keeps the torrent seeding from the original path while making the file appear at the library path, so Jellyfin can see it without forcing you to pick between seeding and having a clean library. If the download and library are on different filesystems, hardlinks silently fall back to copies. Ryokan also writes Jellyfin-compatible NFO sidecars so imported series show up with metadata without Jellyfin needing to hit the internet.
  • Talking to Seerr. Ryokan exposes Sonarr v3 and Radarr v3 API compatibility layers so Seerr thinks it's talking to real instances. Both live on the same host and port but under different URL bases (/api/v3 for Sonarr, /radarr/api/v3 for Radarr) because Seerr caps you at two Sonarr slots and two Radarr slots, so the two Ryokan shims have to coexist on one process.
  • Monitoring and library management. Shows can be monitored in any of five Sonarr-style modes (all, future episodes, missing, existing, or none) and overridden per-episode. A per-series "allow upgrades" toggle lets you opt individual titles out of the upgrade sweep without turning the sweep off globally, useful for shows where you've already picked a release you like and don't want second-guessed.
  • Blocklist. If a grab turns out to be the wrong release, a broken encode, or just something you decided against, blocklisting it means the upgrade sweep and RSS sync will skip it forever. Can be triggered from the Downloads page or from an episode's grab history.
  • Auth. Cookie sessions, bcrypt passwords, first-run setup flow to create the admin account. Login timing is equalized so failed attempts against a nonexistent username take the same ~50ms as failed attempts against a real one (see the advanced technical section, there's a subtlety there). CSRF middleware protects the login and setup POSTs.
  • Settings and observability. Everything configurable (qBit and Jellyfin connections, Custom Formats, preferred and blocked release groups, quality targets, media root) lives in one Settings area. A System page shows live logs, lists every scheduled task with a force-run button, and exposes RSS grab history and a debug tab. There's also a Swagger UI at /api-docs covering every endpoint the web UI actually uses, which has been surprisingly useful for debugging my own code.

Future Feature List

  • Private tracker support (issue #28). Nyaa is great for weeklies and most finished series, but for older or niche content, private trackers have encoders who release exclusively there and nowhere else. The plan is torznab/newznab client support plus autobrr push-target integration, which lets existing tracker tooling push releases into Ryokan without Ryokan needing a bespoke scraper for each tracker. A direct AnimeBytes API integration is on the table but deferred. Torznab covers the case first, then the direct integration if it's worth the extra surface area.
  • Manga, light novel, and web novel support (issue #25). Suwayomi handles manga well but nothing really handles LNs and WNs. The ebook automation space (LazyLibrarian, the various Readarr forks) is fragmented and mostly abandoned. The goal is one unified interface for anime, manga, manhwa/manhua, LNs, and WNs, plugged into the same scheduled-task and post-processing infrastructure Ryokan already has, with source-aware scoring adapted per media type.
  • Notifications. Discord webhooks for completed downloads, failed grabs, and new releases matching a tracked series. Not technically hard, just haven't gotten to it.

Basic Technical Features

  • HTTP routing and middleware. Axum handles the web server, with middleware chains for auth, CSRF, and API-key verification on the Sonarr/Radarr shims. Static files served directly.
  • Server-rendered templates. Askama compiles all templates into the binary at build time with base layout inheritance. This means there's no separate template directory to ship, and a typo in a template is a compile error rather than a runtime 500.
  • Database layer. sqlx against SQLite, covering users, sessions, config, library, grab history, per-episode state, Custom Formats, classification cache, artwork cache, and scheduled task runs. Migrations live in code as idempotent ALTER TABLE ... ADD COLUMN statements that silently ignore "already exists" errors. It's a pattern I picked up from a few other self-hosted projects and it means I never deal with migration file ordering.
  • Auth. bcrypt password hashing, session tokens in SQLite, cookie-based sessions, route-gating middleware that redirects unauthenticated requests to /login. First-run setup at /setup wraps the whole flow.
  • Forms. Settings, setup, and per-page configuration forms with server-side validation and inline error feedback.
  • Docker. Multi-stage Dockerfile (Rust build stage against the official toolchain image, runtime stage on minimal Debian) and a Compose file with the persistent volume layout documented. PUID and PGID environment variables so imported files land with the right ownership for the rest of a user's media stack.
  • CI. GitHub Actions runs cargo build and cargo test on every push and PR to main. A separate workflow builds and publishes multi-arch (amd64 + arm64) Docker images, so the same tag works on anything from a Raspberry Pi to a server.

Advanced Technical Features

  • Three metadata providers, one internal model. AniList (GraphQL) is the primary source. When it 403s or rate-limits, the code falls back to MAL via Jikan (REST) and then Kitsu (JSON:API). Three different API shapes get unified into one internal type so downstream code never has to care which provider it came from. The provider chain is also force-overridable in config so I can pin a specific source when debugging.
  • Six-layer source classification. One orchestrator (source.rs) and six signal modules (source_filename, source_description, source_temporal, source_groups, source_dir, source_ffprobe) vote on a (Source, Resolution, is_remux) verdict. Each layer returns its own confidence, and the aggregator weights them and emits a single confidence score alongside the verdict. Verdicts below the threshold flag needs_review = 1 in the database; a manual override clears that flag and pins the classification against future re-runs of the pipeline. That pin matters. Without it, the upgrade sweep would silently re-grade your manually-corrected episodes on the next pass, and I'd never be able to trust my own data.
  • Custom Formats with a lock-free read path. Custom Formats (user-authored regex-based scoring rules) are edited rarely but evaluated on every scored release, so reading them needs to be fast. I keep compiled CFs in an Arc<RwLock<Arc<Vec<_>>>> on the app state. Handlers clone the inner Arc out under a brief read lock and the scoring hot path is then fully lock-free. On edit, I rebuild the vector off to the side and atomically swap the inner Arc. This also means CF edits take effect for everything currently in-flight without any explicit cache invalidation.
  • The Sonarr/Radarr compatibility layer. The main challenge here was that Seerr allows at most two Sonarr servers and two Radarr servers. That means both of Ryokan's shims have to coexist on one host and one port. I handled that by mounting the Sonarr shim at the canonical /api/v3 prefix and the Radarr shim at /radarr/api/v3, then having both shims sit outside the cookie-auth layer that protects the web UI and use their own query-string API-key middleware instead. The translation layer between Seerr's "add this series" requests (which come in with TMDB IDs) and Ryokan's internal AniList IDs runs through a separate anibridge service backed by a mapping dataset that gets refreshed daily.
  • Supervised background tasks. Seven background loops (RSS sync, post-processing, cleanup, library reclassification, metadata refresh, upgrade search, anibridge refresh) run as tokio::spawned futures wrapped in a supervise() helper that catches panics and join errors, logs them, and respawns the task after 5 seconds. Without that wrapper, a single .unwrap() hitting a weird edge case would silently kill one of the background tasks for the rest of the process lifetime, and I'd only notice days later when shows stopped grabbing. Each task also writes its status and last-run timestamp to a scheduled_task_runs table so the System page can show real health data instead of hoping.
  • Timing-equalized authentication. The obvious implementation of login is "look up the user, if found bcrypt-verify, else return error", which is a timing oracle for username enumeration because the bcrypt path is ~50ms slower than the lookup-failed path. Fix is to bcrypt-verify against a precomputed dummy hash on the missing-user path so both branches take the same time. Subtlety: Rust's LazyLock only computes its inner value on first access, so without explicit initialization the *very first* probe against a bad username would still be a one-shot timing oracle (the lazy init cost of the dummy hash would show up in that one request). I force initialization at startup via warm_timing_equalizer() so that never happens.
  • Auto-generated OpenAPI docs. utoipa derives an OpenAPI schema from handler signatures and serde types, served as JSON at /api-docs/openapi.json with a Swagger UI at /api-docs. Every endpoint the web UI actually calls is documented; the Seerr-facing shim endpoints are too, which has been useful when debugging Seerr's requests without having to strace them.
